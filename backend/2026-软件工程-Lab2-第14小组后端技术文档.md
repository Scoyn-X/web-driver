## 3. 详细设计

### 3.1 API 接口总览

所有接口响应统一用 `Result<T>` 包装（字段：`code` / `data` / `msg` / `requestId` / `traceId`），错误码遵循 **阿里巴巴 A/B/C 三层** 规范：
- `00000` — 成功
- `A****` — 用户端错误（参数、鉴权、资源、业务）
- `B****` — 系统执行错误
- `C****` — 第三方服务错误

完整错误码定义见 `com.jiayuan.boot.common.result.ResultCode`。

#### 3.1.1 用户鉴权（`/api/v1/auth`）

| Method | URL | 鉴权 | 说明 |
|---|---|---|---|
| POST | `/api/v1/auth/register` | 否 | 注册账户，并初始化 1 GB 默认配额 |
| POST | `/api/v1/auth/login` | 否 | 登录，返回 JWT |

##### `POST /api/v1/auth/register`

请求：`RegisterRequestVO`

```json
{
  "nickname": "张三",
  "accountName": "test_user",
  "password": "test1234",
  "email": "test@example.com",
  "accountType": "personal"
}
```

字段说明：

| 字段 | 类型 | 必填 | 约束 |
|---|---|---|---|
| `nickname` | String | 是 | 非空 |
| `accountName` | String | 是 | 非空，全局唯一 |
| `password` | String | 是 | 非空，存储时使用 BCrypt 加密 |
| `email` | String | 是 | 合法邮箱格式，全局唯一 |
| `accountType` | String | 是 | 枚举：`personal` / `work` / `team` |

响应：`Result<Void>`（成功仅返回 `code=00000` 及 `msg="正常响应"`）。

##### `POST /api/v1/auth/login`

请求：`LoginRequestVO`

```json
{
  "accountName": "test_user",
  "password": "test1234"
}
```

响应：`Result<LoginResponseVO>`

```json
{
  "code": "00000",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsImFjY291bnRJZCI6MSwibmlja25hbWUiOiLlvKDkuIkifQ.xxx",
    "userId": 1,
    "accountId": 1,
    "nickname": "张三",
    "accountName": "test_user"
  },
  "msg": "正常响应"
}
```

JWT 签发算法 HS256，过期时间 24h，Claim 中包含 `userId` / `accountId` / `nickname`。前端在后续请求的 `Authorization: Bearer <token>` 头部携带。

**错误码**：

| Code | 常量 | 触发条件 |
|---|---|---|
| A0111 | USERNAME_ALREADY_EXISTS | 注册时 `accountName` 已存在 |
| A0154 | EMAIL_ALREADY_EXISTS | 注册时 `email` 已被注册 |
| A0210 | USER_PASSWORD_ERROR | 登录账户不存在、账户已禁用或密码错误（统一响应以避免账户枚举） |
| A0230 | ACCESS_TOKEN_INVALID | 请求未携带 JWT 或 JWT 过期 / 非法 |

#### 3.1.2 文件管理（`/api/v1/files` + `/api/v1/directories`）

| Method | URL | 鉴权 | 说明 |
|---|---|---|---|
| POST | `/api/v1/files?parentId=` | JWT | 上传文件（multipart；`parentId` 默认 0，可指定子目录 ID，见 §5.4 Bug #1） |
| **GET** | **`/api/v1/files?parentId=`** | **JWT** | **列当前目录下文件 + 子目录（含 breadcrumb 面包屑路径，见 §5.4 R2）** |
| **GET** | **`/api/v1/files/tree`** | **JWT** | **一次性返回当前用户完整文件树（小型网盘用）** |
| GET | `/api/v1/files/{id}/download` | JWT | 下载文件（当前后端流式） |
| DELETE | `/api/v1/files/{id}` | JWT | **删除文件（Bonus §5.3：放入回收站）** |
| POST | `/api/v1/files/{id}/copy` | JWT | 复制文件到指定目录（Bonus §5.2：共享 MinIO 对象、`ref_count++`） |
| PUT | `/api/v1/files/{id}/move` | JWT | 移动文件 / 目录（Bonus §5.2：MinIO 对象不变） |
| GET | `/api/v1/files/search?keyword=` | JWT | （Bonus §5.1）按文件名关键词搜索当前用户的文件（跨目录、仅文件） |
| POST | `/api/v1/directories` | JWT | 新建目录 |
| DELETE | `/api/v1/directories/{id}` | JWT | 删除目录 |
| PUT | `/api/v1/directories/{id}/rename` | JWT | 重命名目录 |
| **GET** | **`/api/v1/directories?parentId=`** | **JWT** | **列指定父目录下的直接子目录（懒加载，每项含 hasChildren 标记；见 §5.4 R1）** |
| **GET** | **`/api/v1/directories/tree`** | **JWT** | **一次性返回当前用户完整目录树（仅文件夹）** |
| **GET** | **`/api/v1/recycle-bin`** | **JWT** | **（Bonus 4.3）当前用户回收站列表（返 RecycleBinItemVO，含可读 path，见 §5.4 R3）** |
| **POST** | **`/api/v1/recycle-bin/{id}/restore`** | **JWT** | **（Bonus 4.3）恢复文件 / 目录（目录递归）** |
| **DELETE** | **`/api/v1/recycle-bin/{id}`** | **JWT** | **（Bonus 4.3）永久删除（真正释放 MinIO + 配额，不可恢复）** |

- **复制**（`copyFile`）：目标目录已存在同名文件时，使用 `generateUniqueFileName` 自动追加 `(1)`、`(2)` …… 序号重命名，如 `report.pdf` → `report(1).pdf` → `report(2).pdf`。扩展名与主名分离，序号永远插在主名尾部。
- **移动**（`moveFile`）：目标目录下若存在同名项目，直接返回 `A0400`，提示 `目标目录下已存在同名文件或目录`；不做自动重命名（语义与「复制新增」不同）。
- 移动目录时，若目标位置是该目录的后代，返回 `A0400`，消息 `不能将目录移动到其子目录下`。
- 目录移动后递归更新所有子项的 `full_path`（祖先 ID 链），保证后续权限判断与路径展示正确；MinIO 对象因此完全不需要重写，详见 §4.1。

**列表响应信封（§5.4 R2 起）**：`GET /api/v1/files?parentId=X` 响应由裸 `List<FileInfoVO>` 改为 `FileListVO`：

```json
{
  "code": "00000",
  "data": {
    "breadcrumb": [
      {"id": 0, "name": "根目录"},
      {"id": 10, "name": "A"},
      {"id": 20, "name": "B"}
    ],
    "items": [/* List<FileInfoVO> */]
  }
}
```
前端可直接渲染面包屑并点击跳转，无需二次请求。

**懒加载 vs 一次性**：目录/文件树均同时提供两套接口共存——`?parentId=X`（按父目录单层懒加载，适合大型网盘）与 `/tree`（一次性完整树，适合小型网盘或需要整体视图的场景）。

**请求 / 响应体**

##### `POST /api/v1/shares`

请求：`ShareCreateRequestVO`

```json
{
  "fileId": 1,
  "accessType": 1,
  "expireDays": 7
}
```

字段：`fileId` 必填（被分享的 `sys_file.id`）；`accessType` 必填（0 = 全公开，1 = 分享码）；`expireDays` 可选，允许 1 / 7 / 30，`null` 表示永久有效。

响应：`Result<ShareInfoVO>`

```json
{
  "code": "00000",
  "data": {
    "id": 500,
    "fileId": 1,
    "fileName": "报告.txt",
    "shareToken": "aa7a08abd2ea417da38dc0c4cac33300",
    "accessType": 1,
    "extractCode": "GYFV",
    "expireTime": "2026-04-21 19:27:20",
    "statusDesc": "有效",
    "createTime": "2026-04-14 19:27:20"
  },
  "msg": "正常响应"
}
```

`extractCode` 仅在 `accessType=1` 时返回，4 位大写字母 / 数字。

##### `GET /api/v1/s/{shareToken}`

响应：`Result<ShareAccessResponseVO>`

```json
{
  "code": "00000",
  "data": {
    "fileName": "报告.txt",
    "fileSize": 58,
    "fileSizeFormatted": "58 B",
    "mimeType": "text/plain",
    "isDirectory": false,
    "requireExtractCode": true,

```

```json
  "code": "00000",
  "data": {
    "downloadUrl": "http://localhost:9000/jiayuan-boot/20260414/abc123.txt?X-Amz-Algorithm=AWS4-HMAC-SHA256&...&X-Amz-Expires=300&X-Amz-Signature=...",
    "fileName": "报告.txt"
  }
}
```

前端持此 URL 直接请求 MinIO 下载。URL 5 分钟内有效。受 MinIO Java SDK 8.5.10 的限制，URL 未携带 `response-content-disposition`，默认下载文件名可能是 object key（`abc123.txt`），由前端通过 `<a download={fileName}>` 属性控制真实文件名，详见 §4.4。

> **设计理由**：目录结构仅存 DB，MinIO object key 与目录解耦。MinIO 中永远是扁平的 `yyyyMMdd/UUID`，对应 Lab2 PDF 2.2.2「方式二」。优势：移动目录 O(N) 仅更新 DB 的 `full_path`，无需重写 MinIO 对象；代价：MinIO 控制台看不到目录树。详细权衡见 §4.1。

> - `uk_share_token`：匿名访问 `/s/{token}` 通过 token 直查，必须唯一且快速

## 4. 关键决策与权衡

### 4.1 目录实现方式

对应 Lab2 PDF 2.2.2 思考题，方式选择：

| 方案 | MinIO 形态 | DB 形态 | 优点 | 缺点 |
|---|---|---|---|---|
| ① 0 字节目录对象 | `user_1/docs/` 空对象 | 不存 | 简单直观 | 移动大目录需重写大量 MinIO 对象 |
| ② **只存 DB**（本组采用） | 扁平 `yyyyMMdd/UUID` | `sys_file.parent_id / full_path` 维护树 | 目录移动 O(N) 仅改 DB | MinIO 控制台看不到目录结构 |
| ③ 混合 | Key 含目录前缀 | DB 也存 | 最灵活 | 两边数据一致性维护成本高 |

**本组选择 ② 的核心理由**：Lab2 PDF 思考题明确指出「移动目录在 MinIO 中重写对象 = 性能灾难」。DB 维护纯逻辑目录树，MinIO 仅承担物理存储，职责解耦。`full_path` 字段保存祖先 ID 链（如 `1,5,12`）解决深层目录的父子关系 O(1) 判断。

### 4.2 配额一致性方案

对应 Lab2 PDF 3.1.4.5 数据一致性要求。两种方案比较：

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A. 每次操作后全量 SUM(`file_size`) | 实时计算 | 简单、永远一致 | 用户文件多时 O(N) 慢 |
| B. **增量维护 `used_space` 字段**（本组采用） | 上传 +size、删除 -size | O(1) 快 | 需保证业务原子性，否则漂移 |

本组采用 B：`QuotaServiceImpl.increaseUsedSpace` / `decreaseUsedSpace` 均以原子 SQL 语句 `UPDATE ... SET used_space = used_space + ?` 或 `... = GREATEST(used_space - ?, 0)` 完成，保证并发安全；`checkQuota` + `putObject` + `insert sys_file` + `increaseUsedSpace` 被 `@Transactional` 包裹，任何一步失败即回滚 DB 状态，孤儿 MinIO 对象由补偿任务清理。`GREATEST` 兜底防止计数异常导致出现负数。

**复制文件的配额语义**：即使 Bonus §5.2 的指纹去重让 MinIO 只保留一份物理对象，复制仍按源文件大小调用 `checkQuota` 与 `increaseUsedSpace`，使每个 `sys_file` 记录都独立计入用户逻辑占用，符合 Lab2 PDF 3.1.4.3「用户逻辑占用」原则：用户视角看到的配额消耗与实际 MinIO 物理去重完全解耦。

**删除路径的归还语义**：

- 普通删除（`deleteFileById`）：Bonus §5.3 已将其语义改为「标记入回收站」，**不释放**配额。
- 永久删除（`permanentlyDelete`）：真正减去 `used_space`，当引用计数归零才删除 MinIO 对象。
- 目录递归删除：先汇总所有后代文件 `file_size`，再一次性 `decreaseUsedSpace`，避免多次 SQL 带来的并发窗口。

### 4.3 分享 token 方案：DB 存储 vs 类 JWT 加密

对应 Lab2 PDF 2.2.4.5 思考题。两种方案对比：

| 方案 | 实现 | 优点 | 缺点 |
|---|---|---|---|
| **① DB 存 token（本组采用）** | `sys_share.share_token` = UUID32；访问时查表 | 支持**主动取消**（改 status）、**过期自动失效**（DB 时间判断）、**文件删除联动提示**（查不到就 A0734） | 每次访问一次 DB 查询（可加 Redis 缓存） |
| ② JWT-like 加密 token | 把 fileId/expire/code 签入 token 返给前端 | 无状态、无需查表 | 无法主动取消（token 派发后有效到过期）；token 字符串较长；需保存 server secret |

**选择 ① 的核心理由**：

1. Lab2 PDF 3.1.3.2 第 5 点明确要求「用户可主动取消分享，取消后链接立即失效」。无状态 token 做不到。
2. 要求「被分享的文件已被所有者删除」需返 `A0734`。无状态 token 无法反映文件状态变化。
3. DB 查询开销微小（`uk_share_token` 唯一索引，毫秒级）。

### 4.4 预签名 URL 取舍 + MinIO Java SDK 8.5.10 限制

**设计意图**：Lab2 PDF 2.2.3 倡导用预签名 URL 减轻后端带宽。分享下载场景引入此技术：后端仅校验分享有效性与提取码，成功后生成 MinIO 预签名 URL（5 分钟有效），前端直连 MinIO 下载，文件数据完全不经过后端。

**实施细节**：`MinioFileServiceImpl.getPresignedDownloadUrl(fileId)` 调用 `minioClient.getPresignedObjectUrl(GetPresignedObjectUrlArgs.builder().bucket(...).object(...).method(GET).expiry(5, TimeUnit.MINUTES).build())`。

**已知限制**：原本计划通过 `extraQueryParams("response-content-disposition", "attachment; filename*=UTF-8''<编码文件名>")` 让 MinIO 在返回文件时直接携带 `Content-Disposition` 响应头，保证浏览器以原文件名保存（而非 object key 的 UUID）。但实测 MinIO Java SDK 8.5.10 生成此类 URL 时签名计算与服务端 canonical 请求不一致，导致 `SignatureDoesNotMatch`（相同内容改用 AWS boto3 SDK 生成的签名验证通过）。暂退回无 `extraQueryParams` 的纯净预签名 URL，**下载文件名由前端通过 `<a download={fileName}>` 属性控制**。

**后续优化方向**：升级 MinIO SDK 到 8.5.22+ 或切换 AWS SDK for Java v2 以启用 `response-content-disposition`。

### 4.5 提取码设计

- **长度** 4 位（Lab2 PDF 4–6 位区间下限）
- **字符集** `ABCDEFGHJKMNPQRSTUVWXYZ23456789`（排除易混字符 `0/O`、`1/I/L`），大写字母 + 数字
- **生成** `cn.hutool.core.util.RandomUtil.randomString(chars, 4)`
- **校验** 大小写不敏感（`equalsIgnoreCase`），UX 友好，符合主流网盘（百度网盘、阿里云盘）习惯
- **持久化** 明文存 `sys_share.extract_code`（非用户密码，泄漏影响有限；分享本身有有效期限制兜底）

### 4.6 已知不足 / 未实施安全增强

| 点 | 现状 | 生产级做法 |
|---|---|---|
| 提取码比对 | `equalsIgnoreCase`（非常数时间） | `MessageDigest.isEqual` 防 timing attack |
| 分享码暴力猜 | 无速率限制 | 加 Redis 按 token+IP 计数，超阈值锁定 |
| MinIO 访问凭证 | 默认 `minioadmin/minioadmin` 写在 yml | K8s Secret / 云厂商 KMS |
| 日志脱敏 | 无 | 统一过滤器屏蔽 token / password |

Lab2 范围内接受上述简化，在此列出以便后续迭代时溯源。

## 5. Bonus 功能设计

### 5.1 文件搜索（已实现）

**目标**：对应 Lab2 PDF 3.1.5 第 3 条，支持按文件名关键词搜索**当前用户**网盘中的**文件**（不含目录），跨所有子目录返回匹配列表。

**实现细节**：

- 入口：`GET /api/v1/files/search?keyword=<关键词>`（JWT 保护）
- Service 方法：`FileService.searchFiles(String keyword)`（interface default method）→ `MinioFileServiceImpl.searchFiles`
- 核心查询（MyBatis-Plus）：

  ```java
  sysFileMapper.selectList(
      new LambdaQueryWrapper<SysFile>()
          .eq(SysFile::getUserId, SecurityUtils.getCurrentUserId())   // 用户隔离
          .eq(SysFile::getIsDirectory, 0)                              // 仅文件
          .eq(SysFile::getInRecycleBin, 0)                             // 排除回收站
          .like(SysFile::getOriginalName, keyword.trim())              // 模糊匹配
          .orderByDesc(SysFile::getCreateTime));                       // 最新优先
  ```

- 软删除（`is_deleted=0`）由 MyBatis-Plus `@TableLogic` 自动过滤。
- 空 / 空白关键词 → 直接返空列表，避免误列所有文件。

**关键设计点**：

- **纯加法扩展**：FileService 使用 default method 扩展，不改动既有 `listFiles` / `copyFile` / `moveFile` 逻辑。
- **用户隔离**：必须 `eq(userId, currentUserId)`，否则暴露他人文件，属于安全关键约束。
- **跨目录**：不限制 `parentId`，与「列当前目录」语义区分清晰。

**已知限制**（Lab 范围内接受）：

1. `LambdaQueryWrapper.like(col, keyword)` 不转义 `%` / `_`，用户输入 `100%` 将被解释为 SQL 通配符；生产级需 `keyword.replace("%", "\\%").replace("_", "\\_")`。
2. 未分页；用户文件极多时返回集合较大。

### 5.2 文件指纹去重存储（已实现）

**目标**：对应 Lab2 PDF 3.1.5 第 1 条，相同内容的文件在 MinIO 只物理保存一份，多个 `sys_file` 记录指向同一 MinIO 对象，删除时用引用计数决定何时真正清理。

**实测结果**（3 个用户上传同一内容）：

| 阶段 | sys_file 可见记录 | sys_file_object ref_count | MinIO 物理对象数 |
|---|---:|---:|---:|
| 3 人上传同内容 | 3 | 3 | **1** |
| 用户 B 删除 | 2 | 2 | 1 |
| 用户 A 删除文件 1 | 1 | 1 | 1 |
| 用户 A 删除最后 1 个 | 0 | (行删除) | **0** |
| 上传新文件 | 1 | 1 | 1 |
| 复制到同目录 | 2 | 2 | **1**（无新 MinIO 对象） |

**数据模型扩展**：

1. 新增 `sys_file_object` 表（物理对象去重索引）：

   ```sql
   CREATE TABLE sys_file_object (
       id BIGINT PK AUTO_INCREMENT,
       file_hash VARCHAR(64) NOT NULL,       -- SHA-256 hex, 64 chars
       object_path VARCHAR(500) NOT NULL,    -- MinIO object key (yyyyMMdd/UUID.ext)
       file_size BIGINT NOT NULL,
       ref_count INT NOT NULL DEFAULT 0,     -- 引用计数
       create_time / update_time,
       UNIQUE KEY uk_file_hash (file_hash)
   );
   ```

2. `sys_file` 增加 `file_hash VARCHAR(64)` 字段（软外键指向 `sys_file_object.file_hash`）。

**流程改写**：

- **上传**：
  1. `DigestInputStream` 边读边算 SHA-256（零额外 IO）。
  2. `SELECT * FROM sys_file_object WHERE file_hash = ?`
     - 命中 → `UPDATE sys_file_object SET ref_count = ref_count + 1 WHERE file_hash = ?`；**跳过** `minioClient.putObject`。
     - 未命中 → `putObject` 到 MinIO + `INSERT sys_file_object (hash, path, size, ref_count=1)`。
  3. `INSERT sys_file` 含新 `file_hash`。
- **复制**：完全不操作 MinIO，只 `INSERT` 新 `sys_file` + `ref_count++`。
- **删除**：`sys_file` 软删 → `ref_count--`；若 `ref_count = 0` → 真正删 MinIO 对象 + `DELETE sys_file_object`。

**配额语义**（呼应 Lab2 PDF 3.1.4.3「用户逻辑占用」）：

- 即使 MinIO 物理只存一份，每个 `sys_file` 记录都计入对应用户 `used_space`。
- 用户视角：复制文件后自己配额确实占用；删除后自己配额释放；与 MinIO 物理状态解耦。

**并发与事务**：

- `ref_count` 增减用原子 SQL：`UPDATE ... SET ref_count = ref_count + 1 WHERE file_hash = ?`。
- 整个上传 / 删除流程包在 `@Transactional` 内；MinIO 操作失败即回滚 DB 状态（需配合清理孤儿对象的补偿任务）。
- 秒传命中判断需避免竞态（两个用户同时首次上传同一哈希）：`INSERT ... ON DUPLICATE KEY UPDATE ref_count = ref_count + 1` 一条语句完成。

**受影响文件清单**：

- 新增：`sql/mysql/sys_file_object.sql`、`SysFileObject.java`、`SysFileObjectMapper.java`
- 修改：`sys_file.sql` + `SysFile.java` 加 `file_hash`，`MinioFileServiceImpl.uploadFile` / `.copyFile` / `.deleteFileById` 改写核心逻辑。

**实施细节（代码落点）**：

- `com.jiayuan.boot.system.oss.model.entity.SysFileObject` + `SysFileObjectMapper`。
- `MinioFileServiceImpl.uploadFile`：SHA-256 计算后先查 `sys_file_object`，命中则跳过 MinIO 上传；未命中则插入物理对象记录。并发上传同一哈希以 `try-catch DuplicateKeyException` 回退为 increment。
- `MinioFileServiceImpl.copyFile`：直接复用源 `filePath/storedName/fileHash`，`ref_count++`，**0 次** MinIO 操作。
- `MinioFileServiceImpl.moveFile`：文件内容不变，MinIO 对象原地不动，仅更新 DB `parent_id/full_path`。
- `MinioFileServiceImpl.deleteFileById`：经 `decrementRefOrRemove` helper；归零时才真正删 MinIO + `sys_file_object`。
- **向后兼容**：`SysFile.fileHash` 为 `null` 的历史数据（Lab1 遗留）走 legacy 路径按 `filePath` 直删，不破坏旧记录。

### 5.3 回收站（已实现）

**目标**：对应 Lab2 PDF 3.1.5 第 2 条，删除文件不立即彻底移除而是进入「回收站」，用户可恢复或永久删除。

**实测结果**：

| 场景 | listFiles | 回收站 | MinIO 物件数 | 配额 |
|---|---:|---:|---:|---|
| 初始上传 2 文件 | 2 | 0 | 2 | 40 B |
| 删除文件 A | 1 | 1 | **2**（保留） | **40 B**（不变） |
| 恢复 A | 2 | 0 | 2 | 40 B |
| 再删除 A + 永久删除 | 1 | 0 | **1**（清理） | **20 B**（释放） |
| 删除目录 my-dir | — | 父+子 2 行 `in_recycle_bin=1` | — | — |
| 恢复 my-dir | — | 父+子 都 `in_recycle_bin=0` | — | — |

**数据模型扩展**：`sys_file` 增加 `in_recycle_bin TINYINT DEFAULT 0`（0 = 正常，1 = 在回收站）。

**API 扩展**：

| Method | URL | 说明 |
|---|---|---|
| GET | `/api/v1/recycle-bin` | 当前用户回收站中的文件列表 |
| POST | `/api/v1/recycle-bin/{id}/restore` | 恢复文件（`in_recycle_bin=0`） |
| DELETE | `/api/v1/recycle-bin/{id}` | 永久删除（`is_deleted=1` + 删 MinIO + 若启用 §5.2 则 `ref_count--`） |
| DELETE | `/api/v1/recycle-bin` | 清空回收站（可选） |

**流程改写**：

- `DELETE /api/v1/files/{id}` 的语义改为仅标记 `in_recycle_bin=1`，**不再**立即删 MinIO 对象。
- `listFiles` / `copyFile` / `moveFile` / `searchFiles` 查询都要加 `in_recycle_bin=0` 过滤（否则显示回收站文件）。
- `ShareServiceImpl.getShareByToken`：若文件 `in_recycle_bin=1`，视同已删除，返 `A0734 SHARE_FILE_DELETED`（分享者看到「文件已删」而不会访问到回收站文件）。

**配额交互**（呼应 Lab2 PDF 3.1.4「用户逻辑占用」原则）：

- 回收站中的文件**继续占用** `used_space`（可随时恢复）。
- 永久删除时才 `decreaseUsedSpace`（配额释放）。
- 恢复时不改配额（一直占用中）。

**目录回收**：递归 mark 子节点 `in_recycle_bin=1`（DFS 按 `parent_id`）；恢复时对称递归还原。

**受影响文件清单**：

- 修改：`sys_file.sql` + `SysFile.java` 加字段，`MinioFileServiceImpl.deleteFileById` 改语义，`listFiles` / `copyFile` / `moveFile` / `searchFiles` 加过滤。
- 新增：`RecycleBinController`、`RecycleBinService` + impl。

**实施细节（代码落点）**：

- `sys_file` 新增 `in_recycle_bin TINYINT` + 复合索引 `(user_id, in_recycle_bin)`。
- `MinioFileServiceImpl.deleteFileById`：语义变更为「标记入回收站」（目录递归）；**不再**删 MinIO、**不再**释放配额。
- `listFiles` / `searchFiles` / `validateFileOwnership` / `validateTargetDirectory` 加 `in_recycle_bin=0` 过滤。
- `FileService` 新增 3 个 default method：`listRecycleBin` / `restoreFromRecycleBin` / `permanentlyDelete`。
- `MinioFileServiceImpl.permanentlyDelete` 复用 §5.2 的 `decrementRefOrRemove`：ref 归零才真正删 MinIO。
- `RecycleBinController` 提供 3 个端点：`GET / POST restore / DELETE`，详见 §3.1.2 API 表。
- `ShareServiceImpl.getShareByToken / getDownloadUrl` 对回收站文件视同已删除，返 `A0734 SHARE_FILE_DELETED`。

### 5.4 前端联调 V2 优化（4 项 + 2 bug）

Bonus 实现后前端联调，发现几项 UX 可优化点与真实 bug。本节记录设计与实施要点，代码散落在 oss / share 两个模块。

#### R1 — 目录树懒加载端点

**动机**：初期 `/api/v1/directories/tree` 一次性返回完整目录树，大型网盘卡顿（几千目录 × 完整递归 = 几十 KB JSON + 前端渲染压力）。

**实现**：新增 `GET /api/v1/directories?parentId=X` 返回该父目录下的直接子目录。新 `DirectoryNodeVO` 字段：`id / name / parentId / hasChildren`。`hasChildren` 用一次 `SELECT DISTINCT parent_id FROM sys_file WHERE parent_id IN (...) AND is_directory=1 AND in_recycle_bin=0` 批量探测（避免 N+1），前端据此决定展开图标是否显示。

**保留**：既有的 `/api/v1/directories/tree` 一次性接口不删——小型网盘仍好用。两套共存。

#### R2 — 文件列表响应改为 FileListVO（含 breadcrumb）

**动机**：前端文件列表需要展示当前目录路径的 breadcrumb 并支持点击跳转。

**实现**：`GET /api/v1/files?parentId=X` 响应由裸 `List<FileInfoVO>` 改为 `FileListVO { breadcrumb: List<BreadcrumbItemVO>, items: List<FileInfoVO> }`。`breadcrumb` 通过解析父目录的 `full_path` ID 链 + 一次 `selectBatchIds` 批量查名拼装；根项固定为 `{id: 0, name: "根目录"}`。总往返 ≤ 2 次。

**破坏性变更**：裸 list 响应形态改了，同 PR 内更新 `ShareFlowIntegrationTest` 对 `data.items` 取值。

#### R3 — 回收站瘦身 + 人类可读 path

**动机**：回收站原返回 `FileInfoVO`，带了 `fileUrl`、`mimeType`、`parentId`、`fullPath`（ID 链）等对前端无用/看不懂的字段；同时前端希望直接看到「文件原位置」。

**实现**：新 `RecycleBinItemVO { id, originalName, path, fileSize, isDirectory, deletedAt }`。`path` 如 `/A/B/baz.txt`，由 `full_path` ID 链 + 一次批量查名拼装。`deletedAt` 取 `sys_file.update_time`（即 `in_recycle_bin` 翻转时刻）。祖先名缺失（历史脏数据/被独立永久删）用 `?` 占位。

#### R4 — 取消分享改为逻辑删除

**动机**：原 `cancelShare` 设 `status=1`，列表仍返回 `statusDesc="已取消"`。前端希望取消后记录消失。

**实现**：`cancelShare` 改为 `sysShareMapper.deleteById(id)` 触发 `BaseEntity.@TableLogic`（写 `is_deleted=1`）。MyBatis-Plus 在后续 `listMyShares` / `findValidShare` 查询时自动过滤；无需改 SQL。附带清理：删除 `STATUS_CANCELLED` 常量、`resolveStatusDesc` 的「已取消」分支、`findValidShare` 的 cancelled 检查（@TableLogic 已处理）。

**行为变更**：
- 重复取消现在抛 `SHARE_NOT_FOUND`（原幂等静默返回）；前端按钮取消后会消失，可接受。
- 用旧 token 访问已取消分享：原 `A0733 SHARE_CANCELLED` → 现 `A0731 SHARE_NOT_FOUND`。`ResultCode.SHARE_CANCELLED` 枚举常量保留避免破坏潜在外部消费者。

#### Bug #1 — 上传支持 parentId + 重名校验按父目录隔离

**症状**：上传端点硬编码 `parentId=0`，加上 `validateNameUnique` 虽然按 parent 过滤但传入 0，用户体验为「整个网盘不允许重名」。

**修复**：
- `FileController.uploadFile` 增加 `@RequestParam(defaultValue="0") Long parentId`
- `FileService.uploadFile` 签名改 `uploadFile(MultipartFile, Long parentId)`；3 个实现（Minio/Local/Aliyun）同步
- `MinioFileServiceImpl.uploadFile` 调 `validateTargetDirectory` 校验父目录，用真实 `parentId` 做 `validateNameUnique` + `sys_file.parentId` + `full_path` 拼接
- 附带修：`DirectoryServiceImpl.validateNameUnique` / `MinioFileServiceImpl.moveFile` 重名校验 / `generateUniqueFileName` 补上 `.eq(SysFile::getInRecycleBin, 0)`，避免回收站里的同名项干扰正常操作

#### Bug #2 — 分享列表反映被删文件状态

**症状**：被分享文件删除（入回收站 or 永久删）后，`GET /api/v1/shares` 仍返 `statusDesc="有效"`，前端看不出已不可访问。

**修复**：
- `listMyShares` 的 `selectBatchIds` 结果从 `Map<Long, String>` 改为 `Map<Long, SysFile>`（带回 `in_recycle_bin`）
- `resolveStatusDesc` 加入 `fileAvailable` 入参，优先级：**已过期 > 文件已删除 > 有效**
- `statusDesc` 新增「文件已删除」分支

### 7.2 测试工程实践要点

为便于后续迭代维护，此处记录在编写过程中沉淀的两条通用约定：

1. **`LambdaQueryWrapper` / `LambdaUpdateWrapper` 的 entity 元信息初始化**：离线单元测试不加载 Spring 上下文，MyBatis-Plus 的 `TableInfo` 不会自动注册，导致 method reference 解析失败。统一做法是在测试类 `@BeforeAll` 中显式初始化：

   ```java
   @BeforeAll
   static void initLambda() {
       MapperBuilderAssistant assistant = new MapperBuilderAssistant(new MybatisConfiguration(), "");
       TableInfoHelper.initTableInfo(assistant, SysFile.class);
       TableInfoHelper.initTableInfo(assistant, SysFileObject.class);
       // 凡测试中使用到的实体均需注册
   }
   ```

2. **`MinioFileServiceImpl` 的非 final 依赖注入**：`minioClient` / `bucketName` / `endpoint` 等字段对 `@InjectMocks` 注入不稳定，统一在 `@BeforeEach` 中通过 `@Data` 生成的 setter 显式注入：

   ```java
   @BeforeEach
   void setUp() {
       service.setBucketName("test-bucket");
       service.setEndpoint("http://localhost:9000");
       service.setMinioClient(minioClient);
   }
   ```

具体模板可参考 `MinioFileServiceImplRecycleBinTest` / `ShareServiceImplTest`。
